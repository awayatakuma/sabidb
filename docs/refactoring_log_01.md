# Rustデータベース開発：Mutexの「聖域」と「解放」

sabidbの並列処理リファクタリングを通じて見えてきた、Rustにおける「守るべき場所」と「攻めるべき場所」の境界線についてまとめます。

---

## 1. 背景：Javaからのポーティングという「大いなる航海」

sabidbはJavaで書かれたエデュケーショナルなDBエンジンのポーティングから始まりました。Javaの `synchronized` をRustで最も安全に再現する手段は、すべてを `Arc<Mutex<T>>` で包むことです。

開発初期において、**「まず動くものを作る（完成させる）」** ことは、並列性能よりも遥かに重要でした。型システムが厳しいRustにおいて、保守的にロックをかける選択は、複雑なDBロジックを迷子にさせないための「道標」として非常に合理的だったのです。

---

## 2. Mutexの「聖域」：なぜそこは外さなかったのか？

リファクタリングを終えた今も、Mutexが居座り続けている場所があります。そこは性能よりも「厳格な順序」が支配する聖域です。

### ● LogManager (ログの追記)
ログは「起きたことの記録」であり、その順番（LSN: Log Sequence Number）が入れ替わることは許されません。

```text
【ログ追記のAA】
   [Tx 1] [Tx 2] [Tx 3]
      \      |      /
       \     |     /
    +-------------------+
    |   GLOBAL MUTEX    | <--- ここは一本道でなければならない
    +-------------------+
             |
             V
    [Log Page (LSN: 10)]
    [Log Page (LSN: 11)] (追記)
```

### ● LockTable (トランザクション排他制御)
「誰がどのデータをロックしているか」という管理表は、データベースの整合性の最後の砦です。ここを中途半端に開放すると、デッドロック検知や排他制御が壊れ、データが破壊されます。

---

## 3. Mutexの「解放」：新しく加えた並列の力

一方で、今回「解放」に成功した箇所では、新しい武器を導入しました。

| コンポーネント | 以前の状態 | 導入したもの | 効果 |
| :--- | :--- | :--- | :--- |
| **Transaction ID** | `Mutex<i32>` | `AtomicI32` | 採番時のグローバルロックを排除 |
| **FileManager** | `Arc<Mutex<FM>>` | `Internal Mutex` + `FileExt` | 同時読み取り・書き込みをOSレベルで実現 |
| **Schema / Layout** | `Arc<Mutex<T>>` | `Arc<Mutex>` 隠蔽型 | 参照時の `.lock().unwrap()` 地獄から解放 |

### FileManagerの進化（AA）
```text
【Before: Java風保守的ロック】          【After: Rust並列最適化】
      [Thread A]  [Thread B]                [Thread A]  [Thread B]
          |           |                         |           |
    +-----+-----------+-----+             +-----+-----------+-----+
    |     FM (Mutex)        |             |  FM (No Mutex)        |
    |  [File] <--Lock中-->  |             |  [File] <---Concurrent|
    +-----------------------+             +-----------------------+
          (Aが終わるまでBは待機)                 (AとBが同時に読み書き!)
```

## 4. DDL（スキーマ変更）と並行処理についての考察

「Schema や Layout から Mutex を剥がしたら、並行して走る別のトランザクションが ALTER TABLE 等でテーブル定義を変更した時に危険ではないのか？」

これはデータベース設計における非常に重要な問いです。
結論から言えば、**Layout や Schema の内部に Mutex を残しても、この問題は安全には解決できません。**

もしトランザクションAが `TableScan` でデータを読んでいる最中に、トランザクションBが `ALTER TABLE DROP COLUMN` を実行したとします。LayoutがMutexで守られていたとしても、Aがレコードを読む度にロックを解放する設計である以上、読んでいる途中でBがカラムを削除してしまう可能性があります。するとAは「さっきまで存在したカラムがない」という矛盾に直面しクラッシュします。

本格的なRDBMSは、この問題を以下のいずれかで解決しています：
1. **メタデータロック（排他テーブルロック）**: トランザクションがアクセス中のテーブルに対し、他のトランザクションがDDLを実行できないようにトランザクションレベルでロックをかける。
2. **MVCC（多版同時実行制御）によるスキーマ管理**: 古いレイアウトと新しいレイアウトを共存させ、実行中のトランザクションは開始時点の古いレイアウトを参照し続ける。

sabidbのような教育用DB（あるいは多くのシンプルなDB）では、DDLは他のトランザクションと並行して実行されない前提に立つか、将来的にメタデータマネージャ層でのグローバルなテーブルロック機構を導入するのが筋です。

したがって、末端のデータ構造である `Layout` 自体は「ある瞬間における不変のスナップショット」として扱い、そこから `Mutex` を剥がしてデータ読み取り（TableScan）の並列性能を最速にする設計は、本格的なアーキテクチャの観点からも正しい進化の方向と言えます。

## 5. リファクタリングの難所と教訓

今回の作業で最も手強かったのは、Rustの「型の連鎖」でした。

1.  **シグネチャの爆発**: `FileManager` からMutexを外した瞬間、それを利用する `Buffer`, `LogManager` など、全レイヤーの型定義を書き換える必要がありました。
2.  **テストコードの移植**: 内部構造を変えると、テストコードも「別の顔」になります。特に `Layout` や `Schema` のように、頻繁に共有されるデータの構造を変えた際、テストケースが求める整合性を維持するのに苦労しました。
3.  **コメントの価値**: 今回、著者の意図（あえて教科書通りにしない等の注釈）を誤って消しそうになりました。ポーティングプロジェクトにおいて、コメントは「なぜこうなっているか」を教える地図であり、コードそのものと同じくらい大切に扱うべきだと痛感しました。

---

## 6. 結論：完成の後の「翼」

「とりあえずMutex」で守られたコードは、未完成のプロジェクトを完成へと導く「安全なゆりかご」でした。
しかし、Rustという強力な言語で真の性能を発揮するためには、どこかでそのゆりかごを卒業し、**「本当に排他が必要な一点」**までロックを絞り込む必要があります。

今回のリファクタリングで、sabidbは並列クエリを高速にさばくための「翼」を手に入れました。

## 7. 補足：アーキテクチャの進化と Rust の哲学

今回のリファクタリングで発生した細かな疑問点と、その設計意図を記録しておきます。

### 7.1. ライフタイム (`'a`) と所有権の選択
`EmbeddedStatement` 等にライフタイムを導入した理由は、無駄な `Arc` のコピーを避け、**「一時的な借り物」**であることを型システムで明示するためです。
`Arc` は共有所有を可能にしますが、常に参照カウントの操作という小さなオーバーヘッドを伴います。Rust では、生存期間が明確な場合は生の参照（`&`）とライフタイムを使うのが最も効率的かつクリーンな設計とされます。

### 7.2. Arc と Mutex の境界線
- **Arc**: 「みんなで読み取り専用のデータを共有する」ためのポインタ。
- **Mutex**: 「誰か一人が書き換えている間、他の人を待たせる」ための鍵。
`Layout` のように生成後に変更されないデータから Mutex を剥がし Arc だけにすることは、「不要な順番待ち」を排除し、マルチコア CPU の性能をフルに引き出すための必須ステップです。

### 7.3. `create_table_internal` 導入の意図
初期化（`new`）プロセスと公開 API（`create_table`）でロジックを共有しつつ、**「再帰的なロック取得や不完全な状態での公開メソッド呼び出し」**を避けるための定石です。内部的な「核心ロジック」を切り出すことで、安全かつ見通しの良い構造になりました。

### 7.4. テストコードの整理
リファクタリングの過程で、過去のコピペミスによる重複した `assert` ブロックなどを整理しました。検証の網羅性を維持しつつ、保守性の高いテストコードへと磨き上げています。
